# PRD_02: 레시피 생성 (Step 2)

## 개요
1단계에서 인식된 냉장고 재료 목록을 바탕으로, 텍스트 생성 모델을 이용해 사용자가 만들 수 있는 요리 레시피를 추천한다.

## 목표
- 인식된 재료 목록을 입력으로 받아, 실제로 조리 가능한 레시피(요리명, 재료, 조리 순서)를 생성한다.
- 재료가 부족한 경우에도 대체 가능한 레시피를 제안한다.

## 범위
- 포함: 레시피 생성 프롬프트 설계, OpenRouter API 연동, 레시피 결과 파싱 및 화면 표시
- 제외: 이미지 인식(1단계), 사용자 프로필/레시피 저장(3단계)
- 전제조건: [[PRD_01]]의 재료 인식 결과(JSON)가 입력으로 제공되어야 함

## 사용 모델
- `openai/gpt-oss-20b:free` (OpenRouter, 텍스트 생성)
- 환경변수: `OPENROUTER_API_KEY`, `OPENROUTER_TEXT_MODEL`

## 사용자 플로우
1. 1단계에서 전달받은 재료 목록(JSON)을 프롬프트에 포함해 텍스트 모델에 요청한다.
2. 모델이 재료를 활용한 레시피 후보(예: 2~3개)를 생성한다.
3. 각 레시피는 요리명, 필요 재료(보유/부족 구분), 조리 순서, 예상 조리 시간을 포함한다.
4. 사용자에게 레시피 후보 목록을 보여주고, 하나를 선택하거나 재생성을 요청할 수 있게 한다.

## API 요청 형식 (예시)
```json
{
  "model": "openai/gpt-oss-20b:free",
  "messages": [
    {
      "role": "user",
      "content": "다음 재료로 만들 수 있는 요리 2~3가지를 추천해줘. 각 요리마다 요리명, 필요 재료(보유/부족 구분), 조리 순서, 예상 조리시간을 알려줘. 결과는 JSON 배열로 줘.\n\n보유 재료: 계란, 우유, 양파"
    }
  ]
}
```

## 출력 데이터 구조 (예시)
```json
{
  "recipes": [
    {
      "title": "양파 계란볶음",
      "ingredients_have": ["계란", "양파"],
      "ingredients_missing": ["대파"],
      "steps": ["양파를 채썬다", "계란을 풀어 팬에 볶는다", "..."],
      "estimated_time_minutes": 15
    }
  ]
}
```

## 프롬프트 설계 원칙
- 응답 형식을 JSON으로 고정하여 파싱 안정성 확보 (모델이 자유 서술로 답하면 파싱 실패 가능 → few-shot 예시 또는 JSON schema 명시)
- 재료 부족 시 "부족 재료" 필드로 명확히 구분하여 사용자가 장보기 목록으로 활용 가능하게 함

## 에러 처리
- JSON 파싱 실패 시 1회 재요청(형식 강조 프롬프트 추가)
- API 오류(429/504 등) 시 재시도 로직 적용

## 완료 기준 (Acceptance Criteria)
- [ ] 재료 목록 입력 시 레시피 2~3개가 정상 생성됨
- [ ] 레시피에 보유/부족 재료가 구분되어 표시됨
- [ ] 조리 순서가 단계별로 표시됨
- [ ] 생성된 레시피가 3단계(저장)로 전달 가능한 형태로 준비됨
